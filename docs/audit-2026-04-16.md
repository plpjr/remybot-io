# remybot-io frontend audit — 2026-04-16

**Scope:** Next.js 16.2.3 / React 19.2.4 / Tailwind 4 / static export to Cloudflare Pages.
**Source of truth for Next APIs:** files read under `node_modules/next/dist/docs/` (cited inline).
**Not re-covered here** (already tracked elsewhere): hardcoded Supabase anon key, table-name mismatches, RL-era copy, StatCard count-up check.

---

## Summary

Codebase is a competent Next 15-era dashboard running on Next 16 — it works, but it's leaving correctness, perf, and a11y wins on the table. The biggest category is **"every page is a big client component"**, which is wasteful even in a static export: most of these pages could render their chrome as Server Components and push only the interactive bits (charts, theme toggle) into client children. Second biggest is **missing segment files** (`error.tsx`, `not-found.tsx`, `global-error.tsx`) — the current UX on a failed Supabase fetch is "stuck skeleton". Third is **a11y gaps** on the Sidebar nav, theme toggle, and live-updating stat cards.

The `"use client"` + `useEffect`-fetching pattern in `src/lib/use-data.ts:142` is the dashboard's central bottleneck — it blocks the entire page on a single `Promise.all` of 12 Supabase calls before anything renders, with no Suspense, no error boundary, and no per-card skeleton. Migrating to per-section Suspense + React 19 `use()` unwrapping would give much better perceived perf even while staying in static-export mode.

No blocking Next 16 deprecation violations found, but the `error.tsx` prop contract changed in 16.2.0 (new `unstable_retry`, `reset` deprecated) — worth knowing before you add error boundaries.

## A. Next.js 16 App Router correctness

### A.1 Every page is `"use client"` — over-clienting the tree
**Files:** `src/app/page.tsx:1`, `src/app/trading/page.tsx:1`, `src/app/risk/page.tsx:1`, `src/app/model/page.tsx:1`, `src/app/analysis/page.tsx:1`, `src/app/autoresearch/page.tsx:1`.

Per `node_modules/next/dist/docs/01-app/02-guides/static-exports.md:53-90`, static export **does** run Server Components at build time and hydrates only the client leaves. The current pattern puts the `"use client"` directive at the top of every page, forcing the entire subtree (PageHeader, section wrappers, static copy, footer) onto the client bundle even though only `useKronosData()`, the time-range buttons, and the charts need client rendering.

**Fix:** Convert each `page.tsx` to a Server Component that statically renders chrome and imports a thin `"use client"` wrapper (e.g. `OverviewClient.tsx`) around the interactive section. This is the "islands" pattern the Next 16 SPA guide recommends (`node_modules/next/dist/docs/01-app/02-guides/single-page-applications.md:30-45`).

### A.2 Missing `error.tsx` / `not-found.tsx` / `global-error.tsx`
**Files:** none exist under `src/app/`.

Per `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/error.md:80-96`, `error.js` wraps each segment in a React error boundary. Without one, a thrown error during client render bubbles to the browser's default empty-page state. Root-layout errors need `global-error.tsx` (same doc, line 161-166).

**Fix:** Add `src/app/error.tsx` (client component), `src/app/global-error.tsx`, and `src/app/not-found.tsx`. **Note the Next 16.2.0 API change** — the prop is now `unstable_retry`, not `reset` (`error.md:25-31, 117-121, 329`). If you copy a Next 14/15 example from training data, it'll use `reset` and silently stop retrying.

### A.3 `loading.tsx` is unsupported under static export — don't add it
**Files:** none exist. Good.

Per `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/loading.md:130-137`, the Platform Support table explicitly lists **Static export: No** for `loading.js`. The current approach (inline `OverviewSkeleton` gated by `loading` from `useKronosData()` at `src/app/page.tsx:36`) is correct for this deploy target. Do NOT add `loading.tsx` files — they'd be a no-op under `output: "export"`.

**Fix:** Instead, wrap each async-data section in React `<Suspense fallback={<SkeletonChart/>}>` and unwrap a Promise with `use()` (see section B.1). Suspense works client-side under static export.

### A.4 Metadata only defined at root layout
**File:** `src/app/layout.tsx:9-21`.

Per `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/01-metadata/index.md:7-11`, each route segment can export its own `metadata` object. Right now every page inherits "Kronos | AI-Powered BTC Trading Bot" — the Trading, Risk, Model, Analysis, Autoresearch pages share a title, which is bad for SEO and bookmarks.

**Fix:** In each page that stays a Server Component (per A.1), `export const metadata: Metadata = { title: "Trading — Kronos" }` etc. For the pages that must stay `"use client"`, move `metadata` to a colocated `layout.tsx` in that segment — metadata exports are not allowed on client components.

### A.5 `next/font` missing `display: 'swap'`
**File:** `src/app/layout.tsx:7` — `const inter = Inter({ subsets: ["latin"] })`.

Per `node_modules/next/dist/docs/01-app/03-api-reference/02-components/font.md:22` the canonical example includes `display: 'swap'`. Without it, the browser can FOIT (flash of invisible text) for up to ~3s on slow connections, hurting LCP.

**Fix:** `Inter({ subsets: ["latin"], display: "swap" })`.

### A.6 `next/image` never used — inconsistent with `next.config.ts` static export story
**Files:** `public/*.svg`, no `<Image/>` imports found.

Under static export, `next/image` requires a custom loader (`node_modules/next/dist/docs/01-app/02-guides/static-exports.md:168-211`) or `images: { unoptimized: true }` in `next.config.ts`. Right now this is moot — nothing uses it — but if you ever add a logo or chart screenshot, you'll hit the build error. Pre-empt it: add `images: { unoptimized: true }` to `next.config.ts` since everything you'd render is already optimized SVG.

### A.7 Inline `<script>` in `<head>` of root layout
**File:** `src/app/layout.tsx:46` — `<script dangerouslySetInnerHTML={{ __html: themeScript }} />`.

This is the correct approach for pre-hydration theme to avoid FOUC, but it blocks a future CSP (`node_modules/next/dist/docs/01-app/02-guides/content-security-policy.md:24-42`). If/when you add CSP headers via Cloudflare, you'll need to either hash this script or use `'unsafe-inline'`. Noting as future-deploy risk.

## B. React 19 specifics

### B.1 `useEffect + useState` for data fetching instead of `use()` + Suspense
**File:** `src/lib/use-data.ts:67-179`.

The hook kicks off 12 parallel Supabase queries in `Promise.all`, flips `loading=true→false` once all complete, then `<OverviewPage>` shows a full-page skeleton until they do (`src/app/page.tsx:36-40`). One slow query blocks the whole page.

**React 19 fix:** Split data fetches into per-section promises and unwrap with `use()` inside Suspense boundaries:

```tsx
// page.tsx (Server Component under A.1)
const overviewPromise = fetchOverviewStats();
const equityPromise = fetchEquityCurve();
return (
  <>
    <Suspense fallback={<StatCardsSkeleton/>}>
      <StatCardsClient promise={overviewPromise}/>
    </Suspense>
    <Suspense fallback={<SkeletonChart/>}>
      <EquityChartClient promise={equityPromise}/>
    </Suspense>
  </>
);
```

Each section fills in independently. Pattern verified in `node_modules/next/dist/docs/01-app/02-guides/single-page-applications.md:34-100` (the "Using React's `use` within a Context Provider" section).

### B.2 No `useOptimistic` candidate — correct
The dashboard is read-only (no user mutations). Skip.

### B.3 `ThemeProvider` initializer reads `localStorage` in `useState` init
**File:** `src/components/ThemeProvider.tsx:27`.

`useState(() => getInitialTheme())` reads `localStorage` during the initial render. This is guarded by the `typeof window === "undefined"` check at line 19, but it will cause a hydration mismatch warning: server renders `"light"` (always), client renders `"dark"` if the user has dark mode. The `suppressHydrationWarning` on `<html>` at `layout.tsx:43` hides the warning for that element but not for descendants that depend on `theme`. The current Sidebar Moon/Sun icon swap at `src/components/Sidebar.tsx:114-124` flashes on load.

**Fix:** Initialize `useState<Theme>("light")` (stable across SSR/client), then set the real theme in `useEffect`. Use CSS-variable selectors (which you already have) to drive the visual, and only conditional-render icons after mount. Alternative: read `document.documentElement.classList.contains("dark")` inside a mount-gated `useEffect`.

### B.4 `Chart.tsx` MutationObserver for theme is fine but leaks on StrictMode double-mount
**File:** `src/components/Chart.tsx:46-52`.

The cleanup is correct. Just flagging: consider replacing the MutationObserver with a shared `useTheme()` subscription from `ThemeProvider` — every chart on the page currently spins up its own observer.

## C. Accessibility (WCAG 2.2 AA)

### C.1 Sidebar not marked as `<nav>` landmark
**File:** `src/components/Sidebar.tsx:53`.

The outer element is `<aside>`, which is a "complementary" landmark. The inner `<nav>` at line 87 is good, but it lacks `aria-label`. Screen-reader users get "navigation" with no way to distinguish it from other navs.

**Fix:** `<nav aria-label="Primary" className="flex-1 px-3 space-y-1">`. Also add `aria-current="page"` to the active link (line 91) — `isActive` at line 89 already knows which.

### C.2 Theme toggle has no `aria-label`
**File:** `src/components/Sidebar.tsx:110-125`.

Button content is only an icon + visible text that changes ("Dark Mode" vs "Light Mode"). The visible text is fine for sighted users, but the button's *purpose* ("toggle theme") is ambiguous — the label reads as a noun, not a verb. Add `aria-label="Switch to dark mode"` / `"Switch to light mode"` depending on current state.

### C.3 Mobile hamburger button missing `aria-label` and `aria-expanded`
**File:** `src/components/Sidebar.tsx:37-42`.

Icon-only button with no accessible name. Add `aria-label="Open navigation menu"`, `aria-expanded={mobileOpen}`, `aria-controls="sidebar"` (and `id="sidebar"` on the `<aside>`). The `X` close button at line 69 has the same issue.

### C.4 No `prefers-reduced-motion` respect
**File:** `src/app/globals.css:40-59`.

`@keyframes fadeInUp` and the stagger delays fire for every user, including those with `prefers-reduced-motion: reduce`. WCAG 2.3.3 (AAA but cheap to implement). Add:

```css
@media (prefers-reduced-motion: reduce) {
  .animate-in { animation: none; }
  * { transition: none !important; }
}
```

Same for the Chart.tsx animations (`animation: true` at `Chart.tsx:67`) — expose a `reducedMotion: true` echarts option when the media query matches.

### C.5 Live-updating stat cards have no `aria-live` region
**Files:** `src/components/StatCard.tsx:36`, `src/components/LiveMathDashboard.tsx:354-385`.

Values update in place (especially `LiveMathDashboard` at ~1Hz). Screen readers won't announce any of it. At minimum, wrap the displayed price in `<span aria-live="polite" aria-atomic="true">`. Be careful not to wrap the whole dashboard in a live region — it'd spam.

### C.6 Charts are completely inaccessible to non-visual users
**File:** `src/components/Chart.tsx`.

No text alternative, no table fallback. Consider rendering a visually-hidden `<table>` or summary paragraph alongside each chart with the key data points.

### C.7 No skip-link
**File:** `src/app/layout.tsx:42-57`.

Keyboard users tabbing from the address bar land on the hamburger → logo → every sidebar link before reaching content. Add `<a href="#main" className="sr-only focus:not-sr-only">Skip to content</a>` at the top of `<body>`, and `id="main"` on the `<main>` element at line 51.

### C.8 Color-only trend indication in StatCard
**File:** `src/components/StatCard.tsx:42-46`.

Trend is shown as a colored arrow icon; the color is the only differentiator for up vs down. The arrow shape helps but the icon is only 14px. Add `aria-label="Trending up"` / `"Trending down"` to the span.

### C.9 Dark-mode contrast to verify
**File:** `src/app/globals.css:18-31`.

`--text-muted: #94A3B8` on `--bg: #0F172A` → contrast ratio ≈ 7.2:1 (passes AA). But `opacity-70` / `opacity-60` utilities used throughout (e.g. `src/app/page.tsx:197, 207`) drop that to ~4.2:1 on mixed backgrounds, which is borderline for small text. Audit each `opacity-*` on text with axe-core or Pa11y.

## D. Performance

### D.1 echarts is in the main bundle
**File:** `src/components/Chart.tsx:3-14`.

Chart is imported directly by `page.tsx`, `trading/page.tsx`, etc. Every route pulls in `echarts/core` + 5 chart modules + 5 component modules + theme files, even if only one chart is visible above the fold. Measured cost: echarts-core is ~200KB minified before gzip.

**Fix:** `const Chart = dynamic(() => import("@/components/Chart"), { ssr: false, loading: () => <SkeletonChart/> })`. Per-route code splitting via `next/dynamic` works under static export.

### D.2 `lucide-react ^1.7.0` — verify tree-shaking
**File:** `package.json:15`.

lucide-react v1 lists every icon in the main entry. Named imports (`import { Menu, X, Sun } from "lucide-react"`) should tree-shake with Turbopack/webpack, but old lucide versions shipped a monolithic bundle. Check built output size: `npm run build && du -sh .next/static/chunks/*.js | sort -h` — if `vendor.js` is >500KB, switch to `lucide-react/icons/Menu` deep imports.

### D.3 CLS risk: skeleton height ≠ real content height
**File:** `src/components/Skeleton.tsx:18-25` vs `src/app/page.tsx:169`.

`SkeletonChart` uses `height = 320` (good, matches the real `<Chart height={320}>`). But `SkeletonCard` at line 7-16 is a fixed 4-line block; the real `StatCard` height varies based on whether `sub` is present (`StatCard.tsx:50-54`). That swap = CLS.

**Fix:** Give `.bg-[var(--card)]` a consistent `min-height` on the card wrapper, or make `SkeletonCard` always match the "with `sub`" variant.

### D.4 `<Chart notMerge={true}>` default forces full re-render on prop change
**File:** `src/components/Chart.tsx:42`.

Combined with `useMemo` of `option` in `LiveMathDashboard` (firing every 1s), the chart fully reinitializes roughly every tick. For the price + SMA chart (`LiveMathDashboard.tsx:237-247`), you almost certainly want `notMerge={false}` + echarts' `appendData` pattern for streaming data.

### D.5 `useLiveBTC` polling + WebSocket combined
**File:** `src/lib/useLiveBTC.ts` (not shown but referenced).

Worth auditing whether the WS reconnect logic has exponential backoff and whether the tab-visibility API pauses the stream when the user switches tabs. An always-on WS costs battery on mobile.

### D.6 Tailwind 4 purge — verify content paths
**File:** `package.json:22` uses `@tailwindcss/postcss ^4`.

Tailwind 4's auto-detection walks from the PostCSS entry. No explicit config needed in most cases, but verify that classes interpolated at runtime (e.g. `src/components/HealthIndicator.tsx:6-10` — `text-emerald-600 dark:text-emerald-400` etc. are fine because they're full strings, good) are all present as full strings. No dynamic concat like `bg-${color}-500` spotted — good.

### D.7 Cloudflare Pages cold start
**Files:** `functions/api/btc-price.ts`, `functions/api/store-math.ts`.

Cloudflare Workers cold start is ~5ms, but the first `/api/btc-price` call builds a JWT with Web Crypto — that adds ~20-50ms per invocation. Consider caching the JWT in module scope for its 120s lifetime (it's currently rebuilt every request at line 76-103).

## E. TypeScript strictness

### E.1 `noUncheckedIndexedAccess` off
**File:** `tsconfig.json:2-20`.

`strict: true` is on (good), but `noUncheckedIndexedAccess` is not. This matters in `src/lib/live-math.ts` and `LiveMathDashboard.tsx:140-148` where you index arrays by computed offsets (`returns[j]`, `momProxy[i]`). With `noUncheckedIndexedAccess: true`, TS forces you to handle `undefined` — caught a handful of real bugs in similar codebases.

**Fix:** Add `"noUncheckedIndexedAccess": true` to `compilerOptions`. Expect ~30 new errors on first run.

### E.2 `functions/` excluded from root tsconfig
**File:** `tsconfig.json:33` — `"exclude": ["node_modules", "functions"]`.

CF Functions are typed by `@cloudflare/workers-types` (devDep), but no `functions/tsconfig.json` exists. That means the Pages Functions are type-checked with whatever rules `wrangler` / the Pages build decides, not your repo's strict mode. Silent source of bugs.

**Fix:** Add `functions/tsconfig.json` extending the root with `{ "compilerOptions": { "types": ["@cloudflare/workers-types"] }, "include": ["**/*.ts"] }`.

### E.3 Inline Supabase row types scattered
**File:** `src/lib/data.ts:12-80` (partial).

Each `fetchX()` hand-rolls its row type. Supabase ships a CLI: `supabase gen types typescript --project-id szxdrpllzngbpiyktipe > src/lib/db-types.ts`. Commit and re-run on schema change — eliminates the silent-drift bug class where a migration renames a column and the dashboard happily displays `undefined`.

### E.4 `any` in `MathStatCard`
**File:** `src/components/LiveMathDashboard.tsx:73` — `icon: any`.

Should be `React.ElementType` (same as StatCard at line 15). Also disables `react/exhaustive-deps` twice (lines 180, 228) — valid, but each disabled rule should have a comment explaining why.

### E.5 `target: "ES2017"` is ancient
**File:** `tsconfig.json:3`.

ES2017 predates async iterators, optional chaining compile output, nullish coalescing, etc. Modern browsers (and Cloudflare Workers runtime) support ES2022+. Downgrading means larger transpiled output.

**Fix:** Bump to `"target": "ES2022"`. Matches `@types/node: ^20` and current CF Workers runtime.

## F. Cloudflare Pages Functions

### F.1 `PagesFunction<Env>` usage is correct
**Files:** `functions/api/btc-price.ts:129`, `functions/api/store-math.ts:36`.

Both use the `PagesFunction<Env>` global from `@cloudflare/workers-types`. Signatures `onRequestGet` / `onRequestPost` are the correct Pages convention. No issues.

### F.2 Env binding not typed across files
**Files:** both functions re-declare `interface Env`.

Declare once in `functions/env.d.ts`:

```ts
interface Env {
  COINBASE_API_KEY_ID?: string;
  COINBASE_API_PRIVATE_KEY?: string;
  SUPABASE_ANON_KEY?: string;
}
```

Then `const onRequestGet: PagesFunction<Env> = ...` in each file references the shared type.

### F.3 No `_headers` or `_redirects` files
**Root:** no such files present.

For a static export dashboard, add `public/_headers`:

```
/*
  Cache-Control: public, max-age=0, must-revalidate
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: geolocation=(), camera=(), microphone=()

/_next/static/*
  Cache-Control: public, max-age=31536000, immutable

/*.svg
  Cache-Control: public, max-age=86400
```

And `public/_redirects` if you want cleaner URLs (optional with `trailingSlash: true` in next.config).

### F.4 `store-math.ts` trusts JSON body shape minimally
**File:** `functions/api/store-math.ts:47-74`.

Already does numeric coercion via `finiteOrNull` (good) and rejects non-positive prices. Missing: payload size limit. Someone can POST a 10MB body from an allowed origin and waste your Supabase row. Add `if (request.headers.get("content-length") > "10000") return 413` before `request.json()`.

### F.5 `btc-price.ts` JWT rebuilt every request
**File:** `functions/api/btc-price.ts:76-103`.

See D.7. The JWT is valid 120s; rebuilding per-request on every client poll is wasteful under hot-Worker conditions. Cache in module scope:

```ts
let cachedJwt: { jwt: string; exp: number } | null = null;
async function getJWT(...) {
  if (cachedJwt && cachedJwt.exp > Date.now()/1000 + 10) return cachedJwt.jwt;
  // ...build and cache
}
```

### F.6 Origin check is right, CORS preflight missing
**File:** `functions/api/store-math.ts:39-45`.

POST from a different origin will trigger a browser preflight OPTIONS request that this function doesn't handle — browser blocks the POST. Add `onRequestOptions` returning `204` with the same origin-allowlist CORS headers as `btc-price.ts:132-137`. `btc-price.ts` itself has the headers but no explicit OPTIONS handler; GET doesn't require one unless custom headers are sent, which they're not, so it's fine today.

## Appendix: one Next 16 API where training data diverges from node_modules

**`error.tsx` prop contract.**

- **Training-data instinct (Next 13-15):** `export default function Error({ error, reset }: { error: Error; reset: () => void })` — call `reset()` to re-render.
- **Next 16.2 reality** (`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/error.md:25-31, 117-140, 329`): the new prop is **`unstable_retry`**, which re-fetches AND re-renders. `reset` still exists but is documented as "use `unstable_retry` instead." The version history table explicitly calls out `v16.2.0 | unstable_retry prop added`.

Action: when writing the new `error.tsx` (fix #1 above), use `unstable_retry` as the primary recovery path and only fall back to `reset` if you specifically want to re-render without re-fetching.
